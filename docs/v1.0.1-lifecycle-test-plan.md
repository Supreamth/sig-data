# V1.0.1 second-release lifecycle test plan

This is a docs-only plan for the next safe CI/CD exercise after the first managed production release. It does not dispatch GitHub Actions, create tags, restart containers, change symlinks, or write release directories.

## Goal

Prove the complete second-release path for `Supreamth/sig-data` now that `v1.0.0` is the first managed production release:

1. make a tiny, reviewable change on a branch,
2. validate it in CI,
3. deploy it to staging first,
4. create a `v1.0.1` tag only from a green protected `main`,
5. run production deploy dry-run for `v1.0.1`,
6. run a real production deploy only after explicit approval,
7. verify that two managed releases now exist, and
8. dry-run rollback from `v1.0.1` back to `v1.0.0`.

## Current baseline

- Canonical repo: `Supreamth/sig-data`.
- Current managed production release: `/opt/sigen-production/releases/v1.0.0`.
- Active production symlink: `/opt/sigen-production/current -> /opt/sigen-production/releases/v1.0.0`.
- First real production deploy: GitHub Actions run `29159055177`.
- First production deploy dry-run: GitHub Actions run `29158829022`.
- Production runner: `sigen-production-srv1698440` with label `sigen-production`.
- Staging first real deploy: GitHub Actions run `29156791754`.
- Rollback caveat: there is no older managed release before `v1.0.0`. Do not invent a rollback path until a second managed release exists.

## Candidate change scope

Keep the `v1.0.1` change intentionally small and low-risk. Good candidates:

- docs-only text update in the public tracker or runbook,
- a harmless UI label/version marker in the dashboard footer/header,
- a non-functional comments/help-text update.

Avoid for this lifecycle test:

- telemetry query changes,
- collector changes,
- InfluxDB/Grafana changes,
- Cloudflare/tunnel changes,
- auth/security changes,
- broad dependency upgrades,
- any staging house-map or EV experiment work.

## Gate 0 — preflight, read-only

Before creating the `v1.0.1` change branch, capture current state:

```bash
git fetch origin --prune --tags
git switch main
git reset --hard origin/main
git status --short --branch
gh run list --repo Supreamth/sig-data --branch main --limit 3
curl -sS -o /tmp/sigen-production-health.json -w 'production=%{http_code}\n' https://sigen.sprees.net/api/health
curl -sS -o /tmp/sigen-staging-health.json -w 'staging=%{http_code}\n' https://sigen-staging.sprees.net/api/health
readlink -f /opt/sigen-production/current
find /opt/sigen-production/releases -maxdepth 1 -mindepth 1 -type d -printf '%p\n' | sort
docker inspect --format '{{.Id}} {{.Name}} {{.Config.Image}} {{.State.Status}} {{.State.StartedAt}}' echarts-dashboard
```

Expected before the second release:

- `main` is clean and CI is green.
- production and staging health return HTTP 200.
- `current` points to `/opt/sigen-production/releases/v1.0.0`.
- only `v1.0.0` exists under `/opt/sigen-production/releases`.

## Gate 1 — branch + PR for the tiny change

Create a normal PR branch from clean `main`:

```bash
git switch -c release/v1.0.1-lifecycle-test
```

Make the tiny selected change, then run local non-deploy validation:

```bash
git diff --check
node --check echarts-dashboard/server.js
node --check echarts-dashboard/public/app.js
python3 -m py_compile *.py
docker compose -f deploy/production/docker-compose.production.yml config >/tmp/production-compose-config-v101.out
curl -sS -o /tmp/sigen-production-health.json -w 'production=%{http_code}\n' https://sigen.sprees.net/api/health
curl -sS -o /tmp/sigen-staging-health.json -w 'staging=%{http_code}\n' https://sigen-staging.sprees.net/api/health
```

Open a PR, wait for CI, and merge only after CI is green. This gate must not run production deployment.

## Gate 2 — staging first

After the PR merges to `main`, deploy the same reviewed `main` state to staging before tagging production.

Dispatch `Deploy Staging` with:

- `confirm`: `DEPLOY_STAGING`
- `dry_run`: `true` first

Verify the staging dry-run changes nothing. Then, after explicit approval, dispatch staging with `dry_run: false`.

Post-staging verification:

```bash
curl -sS -o /tmp/sigen-staging-health.json -w 'staging=%{http_code}\n' https://sigen-staging.sprees.net/api/health
curl -sS -o /tmp/sigen-production-health.json -w 'production=%{http_code}\n' https://sigen.sprees.net/api/health
```

Expected:

- staging health HTTP 200,
- production health HTTP 200,
- production release symlink still points to `v1.0.0`,
- no production workflow was dispatched.

## Gate 3 — tag `v1.0.1` from green main

Only after staging succeeds:

```bash
git fetch origin --prune --tags
git switch main
git reset --hard origin/main
gh run list --repo Supreamth/sig-data --branch main --limit 1
git tag -a v1.0.1 -m "v1.0.1"
git push origin v1.0.1
git ls-remote --tags origin 'refs/tags/v1.0.1*'
```

Verify the tag points to the reviewed `main` commit whose CI passed. Do not move or overwrite release tags.

## Gate 4 — production deploy dry-run for `v1.0.1`

Dispatch `Release Production (manual)` with:

- `mode`: `deploy`
- `release_tag`: `v1.0.1`
- `confirm`: `DEPLOY_PRODUCTION`
- `dry_run`: `true`

Approve the GitHub `production` environment only after confirming the run is dry-run.

Dry-run success evidence should include:

- checkout of `refs/tags/v1.0.1`,
- runner assignment to `sigen-production-srv1698440` or another runner with `sigen-production`,
- script output saying `DRY-RUN`,
- planned release directory `/opt/sigen-production/releases/v1.0.1`,
- planned build/up of only `echarts-dashboard`,
- no release directory created,
- no container restart,
- no symlink change,
- production and staging public health still HTTP 200.

## Gate 5 — real production deploy for `v1.0.1`

Only after dry-run evidence and a distinct explicit approval, dispatch the same workflow with:

- `mode`: `deploy`
- `release_tag`: `v1.0.1`
- `confirm`: `DEPLOY_PRODUCTION`
- `dry_run`: `false`

Post-deploy verification:

```bash
readlink -f /opt/sigen-production/current
find /opt/sigen-production/releases -maxdepth 1 -mindepth 1 -type d -printf '%p\n' | sort
docker inspect --format '{{.Id}} {{.Name}} {{.Config.Image}} {{.State.Status}} {{.State.StartedAt}}' echarts-dashboard
curl -sS -o /tmp/sigen-production-health.json -w 'production=%{http_code}\n' https://sigen.sprees.net/api/health
curl -sS -o /tmp/sigen-staging-health.json -w 'staging=%{http_code}\n' https://sigen-staging.sprees.net/api/health
```

Expected after real deploy:

- `current` points to `/opt/sigen-production/releases/v1.0.1`,
- both `v1.0.0` and `v1.0.1` exist under `/opt/sigen-production/releases`,
- production container was replaced by the managed production compose path,
- production and staging health are HTTP 200,
- staging remains unchanged by production deployment.

## Gate 6 — rollback dry-run to `v1.0.0`

Once `v1.0.1` is live and both release directories exist, prove rollback readiness without changing production:

Dispatch `Release Production (manual)` with:

- `mode`: `rollback`
- `rollback_release`: `/opt/sigen-production/releases/v1.0.0`
- `confirm`: `DEPLOY_PRODUCTION`
- `dry_run`: `true`

Expected:

- rollback validates the target is an existing child directory under `/opt/sigen-production/releases`,
- script prints planned symlink and service restart actions,
- no symlink change,
- no container restart,
- production and staging health still HTTP 200.

A real rollback (`dry_run: false`) should remain a separate incident/release decision and is not part of the normal lifecycle test unless explicitly approved.

## Completion record to add afterward

After all approved gates complete, update:

- `docs/ci-cd-tracker.html` with run IDs, tag target, health results, active symlink, and rollback dry-run result,
- `docs/production-release.md` with a `V1.0.1 execution record`,
- any public handoff/status text that still says `v1.0.1` is only planned.

## Stop conditions

Stop and do not continue to the next gate if any of these occur:

- main CI is not green,
- staging deploy or health fails,
- tag target is not the reviewed green `main` commit,
- production dry-run does not reach the production runner,
- dry-run logs show an unexpected service, container, symlink, path, or environment,
- production or staging health is not HTTP 200,
- `/opt/sigen-production/current` or release directories differ from expectations,
- any secret or `.env` content appears in logs.
